    Table Mappings

    This page provides a comprehensive reference for how MySQL tables are mapped to PostgreSQL tables during the migration process.

    Complete Table Mapping Reference

    Core Entity Mappings

    MySQL Table PostgreSQL Table Migration Type Notes
    Donor_Group (main_group=1) chapter Direct Chapters only
    Donor_Group (main_group=0) chapter_group Direct Chapter groups with org mapping
    Donor person Enhanced Core person entity with UUID conversion
    Member person_chapter_group Many-to-Many Chapter membership relationships

    Contact and Communication

    MySQL Table PostgreSQL Table Migration Type Notes
    Address (primary) address Enhanced Main address to contact table
    Address (non-primary) address Enhanced Additional addresses
    Email email Direct Email data to contact table
    Phone phone Direct Phone data to contact table
    Donor (social fields) social_media Extracted Social media profiles
    Contacted contact_log Enhanced Contact tracking with follow-up features

    Events and Participation

    MySQL Table PostgreSQL Table Migration Type Notes
    Event event Enhanced With chapter mapping
    Participates participates Direct Event attendance with ID mapping

    Financial Data

    MySQL Table PostgreSQL Table Migration Type Notes
    Finance finance Enhanced Gift and pledge information
    Charge transaction Combined Type: 'charge'
    Payment transaction Combined Type: 'payment'

    Relationships and History

    MySQL Table PostgreSQL Table Migration Type Notes
    Family family Direct Family and relationship data
    Schools education Direct Education history
    Job employment Direct Employment history

    Campaign and Volunteer

    MySQL Table PostgreSQL Table Migration Type Notes
    Prospect prospect Direct Campaign prospects
    Volunteer volunteer Direct Volunteer information
    Feasibility_Campaign_Info feasibility_campaign_info Direct Campaign feasibility data

    Recognition and Awards

    MySQL Table PostgreSQL Table Migration Type Notes
    Hall_Of_Fame hall_of_fame Direct Hall of fame records
    Man_Of_Year man_of_year Direct Man of the year awards
    Leadership leadership Direct Leadership roles
    Donor_Status donor_status Direct Donor recognition status
    Key_Players key_players Direct Key player designations

    Migration Type Definitions

    Direct Migration

    Tables that are migrated with minimal structural changes: - Field names may be updated for consistency - Data types are converted to PostgreSQL equivalents - Primary keys converted from INT to UUID - Foreign keys updated to reference new UUIDs

    Enhanced Migration

    Tables that receive significant improvements: - Additional fields for enhanced functionality - Better data validation and constraints - Improved indexing strategies - Enhanced enum types and value validation

    Extracted Migration

    Data extracted from one table into multiple normalized tables: - Improves data chapter and reduces redundancy - Enables better querying and reporting - Maintains referential integrity through foreign keys

    Combined Migration

    Multiple tables combined into a single, more efficient structure: - Reduces complexity and improves performance - Maintains all original data through type fields - Simplifies queries and reporting

    Many-to-Many Migration

    Relationship tables that connect entities: - Properly normalized many-to-many relationships - Enhanced with additional metadata where applicable - Foreign key constraints ensure data integrity

    Field Naming Conventions

    Standardization Changes

    The migration applies consistent naming conventions:

    MySQL Pattern PostgreSQL Pattern Example
    camelCase snake_case firstName → first_name
    PascalCase snake_case LastName → last_name
    lowercase snake_case phonenumber → phone_number
    UPPERCASE lowercase EMAIL → email

    Reserved Word Handling

    PostgreSQL reserved words are handled appropriately:

    MySQL Field PostgreSQL Field Reason
    order sort_order ORDER is reserved in PostgreSQL
    user user_name USER is reserved in PostgreSQL
    group group_name GROUP is reserved in PostgreSQL

    Data Relationship Mapping

    Primary Key Conversion

    All MySQL auto-increment primary keys are converted to PostgreSQL UUIDs:

    -- MySQL
donor_id INT AUTO_INCREMENT PRIMARY KEY

-- PostgreSQL
id UUID PRIMARY KEY DEFAULT gen_random_uuid()

    Foreign Key Relationships

    Foreign key relationships are maintained and enhanced:

    -- MySQL
donor_id INT REFERENCES Donor(donor_id)

-- PostgreSQL
person_id UUID REFERENCES person(id) ON DELETE CASCADE

    Composite Relationships

    Some tables require composite key handling:

    -- Example: Event participation
-- MySQL: (donor_id, event_id)
-- PostgreSQL: (person_id, event_id) with UUIDs

    Special Handling Cases

    Donor_Group Split

    The Donor_Group table is split based on the main_group field:

    -- Chapters (main_group = 1)
INSERT INTO chapter (...)
SELECT ... FROM Donor_Group WHERE main_group = 1;

-- Chapter Groups (main_group = 0)
INSERT INTO chapter_group (...)
SELECT ... FROM Donor_Group WHERE main_group = 0;

    Address Normalization

    Address data is normalized based on importance:

    -- Primary address → contact table
-- Additional addresses → address table

    Transaction Combination

    Financial transactions are combined with type discrimination:

    -- Charges
INSERT INTO transaction (type, ...)
SELECT 'charge', ... FROM Charge;

-- Payments
INSERT INTO transaction (type, ...)
SELECT 'payment', ... FROM Payment;

    Migration Order Dependencies

    The migration order is critical due to foreign key dependencies:

    graph TD
    A[Chapters] --> B[Persons]
    B --> C[Addresses]
    B --> D[Social Media]
    B --> E[Contact Info]
    A --> F[Events]
    B --> F
    F --> G[Participates]
    B --> H[Finance]
    H --> I[Transactions]
    B --> J[Family]
    B --> K[Education]
    B --> L[Employment]
    B --> M[Campaign Data]
    B --> N[Contact Log]

    Dependency Rules

    1. Chapters before Persons: Person records may reference chapters
    2. Persons before Everything Else: Most tables have foreign keys to person
    3. Events before Participates: Participation requires existing events
    4. Finance before Transactions: Transactions reference finance records

    Validation and Integrity

    Post-Migration Validation

    After migration, several validation checks ensure data integrity:

    1. Record Count Validation

    -- Verify all records migrated
SELECT
    'donor' as table_name,
    COUNT(*) as mysql_count
FROM mysql.Donor
UNION ALL
SELECT
    'person' as table_name,
    COUNT(*) as postgres_count
FROM person;

    2. Foreign Key Integrity

    -- Check for orphaned records
SELECT table_name, COUNT(*) as orphaned_records
FROM (
    SELECT 'address' as table_name
    FROM address a
    LEFT JOIN person p ON a.person_id = p.id
    WHERE p.id IS NULL

    UNION ALL

    SELECT 'contact_log' as table_name
    FROM contact_log cl
    LEFT JOIN person p ON cl.person_id = p.id
    WHERE p.id IS NULL

    -- Additional orphan checks...
) orphan_checks
GROUP BY table_name;

    3. Data Quality Checks

    -- Verify data quality
SELECT
    'person' as table_name,
    COUNT(CASE WHEN first_name IS NULL OR first_name = '' THEN 1 END) as missing_first_names,
    COUNT(CASE WHEN last_name IS NULL OR last_name = '' THEN 1 END) as missing_last_names
FROM person;

    This mapping reference ensures that all data relationships are properly maintained during the migration from MySQL to PostgreSQL while taking advantage of modern database features and improved data chapter.
