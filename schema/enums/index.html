    Database Enums

    Overview

    This document provides a complete reference of all enum types defined in the GMS database schema. Enums ensure data consistency, improve query performance, and provide better validation compared to free-text fields.

    Person and Status Enums

    Person Status Enum

    Tracks the current status of a person in the system. 

    CREATE TYPE person_status_enum AS ENUM ('Active', 'In-active', 'Lost', 'Deceased');

    Values: - Active - Person is currently active in the chapter - In-active - Person is temporarily inactive - Lost - Lost contact with the person - Deceased - Person is deceased

    MySQL to PostgreSQL Mapping: | MySQL Value | PostgreSQL Enum | Description | |-------------|-----------------|-------------| | 'A' | 'Active' | Active member | | 'I' | 'In-active' | Inactive member | | 'L' | 'Lost' | Lost contact | | 'D' | 'Deceased' | Deceased member |

    Person Code Enum

    Categorizes the person's relationship to the chapter. 

    CREATE TYPE person_code_enum AS ENUM ('Alumni', 'Undergrad', 'Parent', 'Friend', 'Alumni/Parent');

    Values: - Alumni - Graduate/former member - Undergrad - Current undergraduate member - Parent - Parent of a member - Friend - Friend of the chapter - Alumni/Parent - Person who is both alumni and parent

    MySQL to PostgreSQL Mapping: | MySQL Value | PostgreSQL Enum | Description | |-------------|-----------------|-------------| | 'A' | 'Alumni' | Alumni/graduate member | | 'U' | 'Undergrad' | Undergraduate member | | 'P' | 'Parent' | Parent of member | | 'F' | 'Friend' | Friend of chapter | | 'X' | 'Alumni/Parent' | Alumni who is also a parent |

    Marital Status Enum

    Tracks marital status of individuals. 

    CREATE TYPE marital_status_enum AS ENUM ('Single','Married','Widowed','Divorced','Unknown');

    Values: - Single - Not married - Married - Currently married - Widowed - Spouse is deceased - Divorced - Previously married, now divorced - Unknown - Marital status unknown

    MySQL to PostgreSQL Mapping: | MySQL Value | PostgreSQL Enum | Description | |-------------|-----------------|-------------| | 'S' | 'Single' | Single/unmarried | | 'M' | 'Married' | Currently married | | 'W' | 'Widowed' | Widowed | | 'D' | 'Divorced' | Divorced | | '?' | 'Unknown' | Unknown marital status |

    Chapter and Date Enums

    Year Type Enum

    Defines what type of year is being referenced for the chapter. 

    CREATE TYPE year_type_enum AS ENUM ('Initiation', 'Graduation');

    Values: - Initiation - Year refers to initiation date - Graduation - Year refers to graduation date

    Address and Location Enums

    Address Type Enum

    Categorizes different types of addresses. 

    CREATE TYPE address_type_enum AS ENUM (
  'Home', 'Business', 'Mailing', 'Billing', 'Shipping',
  'Summer Home', 'Winter Home', 'Previous', 'Other'
);

    Values: - Home - Primary residence - Business - Work/business address - Mailing - Preferred mailing address - Billing - Billing address for financial transactions - Shipping - Address for shipping items - Summer Home - Seasonal summer residence - Winter Home - Seasonal winter residence - Previous - Former address - Other - Other address type

    MySQL to PostgreSQL Mapping: | MySQL Value | PostgreSQL Enum | Description | |-------------|-----------------|-------------| | 'Home' | 'Home' | Primary residence | | 'Business' | 'Business' | Work/business address | | 'Summer Home' | 'Summer Home' | Seasonal summer residence | | 'Winter Home' | 'Winter Home' | Seasonal winter residence | | 'Other' | 'Other' | Other address type | | Any unmapped value | 'Mailing' | Default for new address types |

    Geocode Accuracy Enum

    Indicates the accuracy level of geocoded addresses. 

    CREATE TYPE geocode_accuracy_enum AS ENUM (
  'ROOFTOP', 'RANGE_INTERPOLATED', 'GEOMETRIC_CENTER',
  'APPROXIMATE', 'UNKNOWN'

    Values: - ROOFTOP - Exact building location - RANGE_INTERPOLATED - Interpolated from address range - GEOMETRIC_CENTER - Center of geographic area - APPROXIMATE - Approximate location - UNKNOWN - Accuracy unknown

    CREATE TYPE validation_status_enum AS ENUM (

    Values: - VERIFIED - Information has been verified as correct - UNVERIFIED - Information has not been verified - INVALID - Information is known to be invalid - PENDING - Verification is in progress - FAILED - Verification attempt failed

    Database Usage Guidelines

    Best Practices

    1. Always use enum values exactly as defined - Enums are case-sensitive
    2. Handle enum values in application code - Don't rely on database for enum validation alone
    3. Plan for enum expansion - Consider future values when designing application logic
    4. Use descriptive enum values - Values should be self-documenting

    Query Examples

    -- Filter by person status
SELECT * FROM person WHERE status = 'Active';

-- Group by contact method preference
SELECT contact_method, COUNT(*)
FROM contact_preferences
GROUP BY contact_method;

-- Find people by marital status
SELECT full_name, marital_status
FROM person
WHERE marital_status IN ('Married', 'Single');

-- Contact log summary by type
SELECT contact_type, COUNT(*) as contact_count
FROM contact_log
GROUP BY contact_type
ORDER BY contact_count DESC;

    Index Performance

    Enums provide excellent index performance and are stored efficiently:

    -- Efficient indexes on enum columns
CREATE INDEX idx_person_status ON person(status);
CREATE INDEX idx_contact_type ON contact_log(contact_type);
CREATE INDEX idx_address_type ON address(address_type);

    This enum system ensures data consistency, improves query performance, and provides clear data validation throughout the GMS database.

    Migration Mapping Functions

    The following functions help convert MySQL enum values to PostgreSQL enum values during migration:

    Generic Enum Conversion Function

    -- Generic enum conversion function with fallback
CREATE OR REPLACE FUNCTION convert_mysql_enum(
    input_value TEXT,
    mapping JSONB,
    default_value TEXT DEFAULT 'Other'
) RETURNS TEXT AS $$
DECLARE
    result TEXT;
BEGIN
    -- Handle NULL or empty input
    IF input_value IS NULL OR TRIM(input_value) = '' THEN
        RETURN default_value;
    END IF;

    -- Try direct mapping first
    result := mapping->>input_value;

    -- If no mapping found, return default
    IF result IS NULL THEN
        result := default_value;
    END IF;

    RETURN result;
END;
$$ LANGUAGE plpgsql;

    Specific Mapping Functions

    -- Person status mapping
CREATE OR REPLACE FUNCTION map_person_status(mysql_status TEXT) RETURNS person_status_enum AS $$
BEGIN
    RETURN CASE mysql_status
        WHEN 'A' THEN 'Active'::person_status_enum
        WHEN 'I' THEN 'In-active'::person_status_enum
        WHEN 'L' THEN 'Lost'::person_status_enum
        WHEN 'D' THEN 'Deceased'::person_status_enum
        ELSE 'Active'::person_status_enum
    END;
END;
$$ LANGUAGE plpgsql;

-- Person code mapping
CREATE OR REPLACE FUNCTION map_person_code(mysql_code TEXT) RETURNS person_code_enum AS $$
BEGIN
    RETURN CASE mysql_code
        WHEN 'A' THEN 'Alumni'::person_code_enum
        WHEN 'U' THEN 'Undergrad'::person_code_enum
        WHEN 'P' THEN 'Parent'::person_code_enum
        WHEN 'F' THEN 'Friend'::person_code_enum
        WHEN 'X' THEN 'Alumni/Parent'::person_code_enum
        ELSE 'Alumni'::person_code_enum
    END;
END;
$$ LANGUAGE plpgsql;

-- Marital status mapping
CREATE OR REPLACE FUNCTION map_marital_status(mysql_status TEXT) RETURNS marital_status_enum AS $$
BEGIN

        WHEN 'S' THEN 'Single'::marital_status_enum
        WHEN 'M' THEN 'Married'::marital_status_enum

    END;

    Migration Usage Examples

    -- Example usage during data migration
UPDATE person
SET status = map_person_status(mysql_mstatus),
    code = map_person_code(mysql_mcode);

UPDATE family
SET marital_status = map_marital_status(mysql_mar_status);

UPDATE phone_number
SET phone_type = map_phone_type(mysql_ptype);

UPDATE address
SET address_type = map_address_type(mysql_atype);

    Validation Queries for Migration

    -- Check for unmapped values before migration
SELECT
    'person_status' as field_name,
    mysql_mstatus as mysql_value,
    COUNT(*) as count
FROM mysql_general_frat_info_temp
WHERE mysql_mstatus NOT IN ('A', 'I', 'L', 'D')
GROUP BY mysql_mstatus

UNION ALL

SELECT
    'person_code' as field_name,
    mysql_mcode as mysql_value,
    COUNT(*) as count
FROM mysql_general_frat_info_temp
WHERE mysql_mcode NOT IN ('A', 'U', 'P', 'F', 'X')
GROUP BY mysql_mcode

UNION ALL

SELECT
    'marital_status' as field_name,
    mysql_mar_status as mysql_value,
    COUNT(*) as count
FROM mysql_family_temp
WHERE mysql_mar_status NOT IN ('S', 'M', 'W', 'D', '?')
GROUP BY mysql_mar_status

UNION ALL

SELECT
    'phone_type' as field_name,
    mysql_ptype as mysql_value,
    COUNT(*) as count
FROM mysql_phone_temp
WHERE mysql_ptype NOT IN ('Home', 'Work', 'Cell', 'Fax', 'Other')
GROUP BY mysql_ptype

ORDER BY field_name, count DESC;
