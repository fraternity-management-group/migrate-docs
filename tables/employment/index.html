    Employment Table Migration

    Overview

    The Employment table migration transforms the simple MySQL Job table into a PostgreSQL employment tracking system with basic employment information and enhanced data normalization.

    Schema Transformation

    MySQL Source Table: Job

    CREATE TABLE `Job` (
  `donor_id` int(10) unsigned NOT NULL DEFAULT 0,
  `company` varchar(45) NOT NULL DEFAULT '',
  `position` varchar(120) DEFAULT NULL,
  `past_company` varchar(45) DEFAULT NULL,
  `past_position` varchar(120) DEFAULT NULL,
  `category` varchar(45) DEFAULT NULL,
  `address` varchar(255) DEFAULT NULL,
  `phone` varchar(45) DEFAULT NULL,
  `url` varchar(45) DEFAULT NULL,
  `comments` text DEFAULT NULL,
  `job_cat` varchar(45) DEFAULT NULL,
  `last_changed` timestamp NOT NULL DEFAULT current_timestamp() ON UPDATE current_timestamp(),
  PRIMARY KEY (`donor_id`)
);

    PostgreSQL Target Table: employment

    CREATE TABLE IF NOT EXISTS employment (
  person_id UUID PRIMARY KEY REFERENCES person(id) ON DELETE CASCADE,
  company TEXT NOT NULL DEFAULT '',
  position TEXT,
  soc_code VARCHAR(7),
  naics_code VARCHAR(6),
  category VARCHAR(45),
  url TEXT,
  comments TEXT,
  achievements TEXT[],
  responsibilities TEXT[],
  employment_status employment_status_enum,
  work_arrangement work_arrangement_enum,
  start_date DATE,
  end_date DATE,
  last_changed TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

    Field Mappings

    Direct Field Mappings

    MySQL Field PostgreSQL Field Type Change Notes
    donor_id person_id INT → UUID Primary key, references person table
    company company VARCHAR(45) → TEXT Company name
    position position VARCHAR(120) → TEXT Job position/title
    category category VARCHAR(45) → VARCHAR(45) Employment category
    url url VARCHAR(45) → TEXT Company/job URL
    comments comments TEXT → TEXT Additional comments
    last_changed last_changed TIMESTAMP → TIMESTAMP Last modification time

    Fields Not Migrated from MySQL

    The following MySQL fields are not migrated to PostgreSQL:

    • past_company - Historical employment not captured in current schema
    • past_position - Historical employment not captured in current schema
    • address - Handled separately in address table with type 'Business'
    • phone - Handled separately in phone_number table with type 'Work'
    • job_cat - Used as fallback for category if category is NULL

    New PostgreSQL Fields (Set to NULL during migration)

    PostgreSQL Field Type Purpose Migration Value
    soc_code VARCHAR(7) Standard Occupational Classification NULL
    naics_code VARCHAR(6) North American Industry Classification NULL
    achievements TEXT[] Notable achievements NULL
    responsibilities TEXT[] Job responsibilities NULL
    employment_status employment_status_enum Employment status NULL
    work_arrangement work_arrangement_enum Work arrangement type NULL
    start_date DATE Employment start date NULL
    end_date DATE Employment end date NULL

    Data Type Enums

    The PostgreSQL employment table uses the following enum types:

    Employment Status Enum

    CREATE TYPE employment_status_enum AS ENUM (
    'Full-time',
    'Part-time',
    'Contract',
    'Consulting',
    'Retired',
    'Self-employed',
    'Unemployed',
    'Student'
);

    Work Arrangement Enum

    CREATE TYPE work_arrangement_enum AS ENUM (
    'On-site',
    'Remote',
    'Hybrid',
    'Travel',
    'Field-based'
);

    Key Improvements

    Data Normalization

    1. Separated Contact Information

      • Work addresses moved to dedicated address table
      • Work phone numbers moved to dedicated phone_number table
      • Maintains referential integrity with proper foreign keys

    2. Enhanced Data Types

      • UUIDs for better distributed system support
      • TEXT fields for unlimited content length
      • Array types for multi-value fields

    3. Enum Validation

      • Standardized employment status values
      • Consistent work arrangement classifications

    Migration Notes

    Data Quality Considerations

    1. Simple Source Data

      • MySQL Job table contains minimal employment information
      • Many PostgreSQL fields will be NULL after initial migration
      • Future enhancements can populate additional fields

    2. Category Field Logic

      • Uses category field if available
      • Falls back to job_cat field if category is NULL
      • Both fields can contain employment categorization information

    3. Historical Employment

      • past_company and past_position fields not migrated
      • Current schema focuses on current employment only
      • Historical employment tracking would require schema enhancement

    Indexes and Performance

    Primary Indexes

    -- Primary key constraint automatically creates index
-- employment_pkey on employment(person_id)

    Recommended Indexes for Performance

    -- Company-based queries
CREATE INDEX IF NOT EXISTS idx_employment_company ON employment(company);

-- Category-based queries
CREATE INDEX IF NOT EXISTS idx_employment_category ON employment(category);

-- URL-based queries
CREATE INDEX IF NOT EXISTS idx_employment_url ON employment(url) WHERE url IS NOT NULL;

-- Last changed timestamp for audit queries
CREATE INDEX IF NOT EXISTS idx_employment_last_changed ON employment(last_changed);

    Common Queries

    Basic Employment Information

    -- Get all employment records with person details
SELECT
    p.first_name,
    p.last_name,
    e.company,
    e.position,
    e.category,
    e.last_changed
FROM employment e
JOIN person p ON e.person_id = p.id
ORDER BY e.last_changed DESC;

    Company-Based Analysis

    -- Count employees by company
SELECT
    company,
    COUNT(*) as employee_count
FROM employment
WHERE company IS NOT NULL AND company != ''
GROUP BY company
ORDER BY employee_count DESC;

    Employment by Category

    -- Analyze employment by category
SELECT
    COALESCE(category, 'Unknown') as employment_category,
    COUNT(*) as count
FROM employment
GROUP BY category
ORDER BY count DESC;

    Recent Employment Updates

    -- Find recently updated employment records
SELECT
    p.first_name,
    p.last_name,
    e.company,
    e.position,
    e.last_changed
FROM employment e
JOIN person p ON e.person_id = p.id
WHERE e.last_changed >= CURRENT_DATE - INTERVAL '30 days'
ORDER BY e.last_changed DESC;

    Migration Verification

    Data Integrity Checks

    1. Record Count Verification 

      -- Compare record counts between source and target
SELECT 'MySQL Job' as source, COUNT(*) as count FROM Job
UNION ALL
SELECT 'PostgreSQL employment' as source, COUNT(*) as count FROM employment;

    2. Field Population Analysis 

      -- Check field population rates
SELECT
    COUNT(*) as total_records,
    COUNT(company) as company_populated,
    COUNT(position) as position_populated,
    COUNT(category) as category_populated,
    COUNT(url) as url_populated,
    COUNT(comments) as comments_populated,
    ROUND(COUNT(company) * 100.0 / COUNT(*), 2) as company_percentage,
    ROUND(COUNT(position) * 100.0 / COUNT(*), 2) as position_percentage
FROM employment;

    3. Foreign Key Validation 

      -- Verify all employment records have valid person references
SELECT COUNT(*) as orphaned_records
FROM employment e
LEFT JOIN person p ON e.person_id = p.id
WHERE p.id IS NULL;

    Data Quality Improvements

    Validation Rules

    1. Basic Data Validation
    2. Company name should not be empty string if populated
    3. URL format validation for web addresses

    4. Category values should be standardized

    5. Referential Integrity

    6. All person_id values must exist in person table
    7. Foreign key constraints ensure data consistency

    Data Cleanup Opportunities

    1. Company Name Standardization 

      -- Find potential duplicate companies with different formatting
SELECT company, COUNT(*)
FROM employment
WHERE company IS NOT NULL AND company != ''
GROUP BY LOWER(TRIM(company))
HAVING COUNT(*) > 1
ORDER BY COUNT(*) DESC;

    2. Category Standardization 

      -- Analyze category values for standardization opportunities
SELECT category, COUNT(*)
FROM employment
WHERE category IS NOT NULL
GROUP BY category
ORDER BY COUNT(*) DESC;
