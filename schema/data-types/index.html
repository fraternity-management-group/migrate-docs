    Data Type Conversions

    This page details how MySQL data types are converted to their PostgreSQL equivalents during the migration process.

    Primary Data Type Mappings

    Numeric Types

    MySQL Type PostgreSQL Type Notes
    TINYINT SMALLINT 8-bit → 16-bit integer
    TINYINT(1) BOOLEAN MySQL boolean emulation → native boolean
    SMALLINT SMALLINT Direct mapping
    MEDIUMINT INTEGER 24-bit → 32-bit integer
    INT / INTEGER INTEGER Direct mapping
    BIGINT BIGINT Direct mapping
    DECIMAL(p,s) DECIMAL(p,s) Direct mapping with precision
    FLOAT REAL Single precision floating point
    DOUBLE DOUBLE PRECISION Double precision floating point

    String Types

    MySQL Type PostgreSQL Type Notes
    CHAR(n) CHAR(n) Fixed-length character string
    VARCHAR(n) VARCHAR(n) Variable-length character string
    TEXT TEXT Unlimited variable-length text
    TINYTEXT TEXT Small text → unlimited text
    MEDIUMTEXT TEXT Medium text → unlimited text
    LONGTEXT TEXT Large text → unlimited text
    BINARY(n) BYTEA Fixed-length binary → variable binary
    VARBINARY(n) BYTEA Variable-length binary data
    BLOB BYTEA Binary large object

    Date and Time Types

    MySQL Type PostgreSQL Type Notes
    DATE DATE Direct mapping
    TIME TIME Direct mapping
    DATETIME TIMESTAMP Date and time without timezone
    TIMESTAMP TIMESTAMPTZ Date and time with timezone
    YEAR SMALLINT Year stored as integer

    Special Conversions

    MySQL Type PostgreSQL Type Notes
    ENUM('a','b','c') Custom ENUM Type-safe enumeration
    SET('a','b','c') TEXT[] Array of text values
    JSON JSONB Binary JSON for better performance
    AUTO_INCREMENT UUID Sequential integers → UUIDs

    Boolean Type Conversion

    MySQL TINYINT(1) to PostgreSQL BOOLEAN

    MySQL uses TINYINT(1) to emulate boolean values, while PostgreSQL has native boolean support.

    Conversion Logic

    -- MySQL values
TINYINT(1): 0, 1, NULL

-- PostgreSQL conversion
CASE 
    WHEN mysql_field = 1 THEN true
    WHEN mysql_field = 0 THEN false
    ELSE NULL
END

    Example Fields

    Table MySQL Field PostgreSQL Field Conversion
    Donor active TINYINT(1) is_active BOOLEAN 0/1 → false/true
    Address primary TINYINT(1) is_primary BOOLEAN 0/1 → false/true
    Contacted restricted TINYINT(1) is_restricted BOOLEAN 0/1 → false/true

    Primary Key Conversion

    AUTO_INCREMENT to UUID

    All MySQL auto-increment primary keys are converted to PostgreSQL UUIDs for better distributed system support.

    Conversion Process

    -- MySQL
CREATE TABLE donor (
    donor_id INT AUTO_INCREMENT PRIMARY KEY,
    first_name VARCHAR(50)
);

-- PostgreSQL
CREATE TABLE person (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    first_name VARCHAR(50)
);

    ID Mapping Table

    During migration, a mapping table tracks the conversion:

    CREATE TABLE id_mapping (
    mysql_table VARCHAR(50),
    mysql_id INTEGER,
    postgres_id UUID,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

    Foreign Key Updates

    All foreign key references are updated to use the new UUIDs:

    -- MySQL
donor_id INT REFERENCES donor(donor_id)

-- PostgreSQL  
person_id UUID REFERENCES person(id)

    Enumeration Conversion

    MySQL ENUM to PostgreSQL ENUM

    MySQL enum values are converted to PostgreSQL custom enum types for better type safety.

    Person Status Example

    -- MySQL
status ENUM('Active', 'In-active', 'Lost', 'Deceased')

-- PostgreSQL
CREATE TYPE person_status_enum AS ENUM (
    'Active', 
    'In-active', 
    'Lost', 
    'Deceased'
);

status person_status_enum

    Contact Type Example

    -- MySQL
ctype INT  -- 1=Phone, 2=Email, 3=Letter, etc.

-- PostgreSQL
CREATE TYPE contact_type_enum AS ENUM (
    'Phone Call',
    'Email', 
    'Letter',
    'In-Person Meeting',
    'Text Message',
    'Social Media',
    'Video Call',
    'Voicemail',
    'Event Contact',
    'Newsletter',
    'Donation Thank You',
    'Other'
);

contact_type contact_type_enum

    Enum Mapping Logic

    Integer-based enums are mapped using CASE statements:

    CASE mysql_ctype
    WHEN 1 THEN 'Phone Call'::contact_type_enum
    WHEN 2 THEN 'Email'::contact_type_enum
    WHEN 3 THEN 'Letter'::contact_type_enum
    WHEN 4 THEN 'In-Person Meeting'::contact_type_enum
    ELSE 'Other'::contact_type_enum
END

    Date and Time Handling

    Timezone Considerations

    MySQL TIMESTAMP fields are converted to PostgreSQL TIMESTAMPTZ to properly handle timezones.

    -- MySQL
created_timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP

-- PostgreSQL
created_at TIMESTAMPTZ DEFAULT NOW()

    Invalid Date Handling

    MySQL allows invalid dates like '0000-00-00', which PostgreSQL rejects:

    -- Conversion logic
CASE 
    WHEN mysql_date = '0000-00-00' THEN NULL
    WHEN mysql_date IS NULL THEN NULL
    WHEN mysql_date < '1900-01-01' THEN NULL
    ELSE mysql_date::DATE
END

    Date Range Validation

    PostgreSQL is stricter about date ranges:

    -- Handle edge cases
CASE 
    WHEN mysql_year < 1900 THEN NULL
    WHEN mysql_year > 2100 THEN NULL
    ELSE make_date(mysql_year, 1, 1)
END

    Text and Character Encoding

    Character Set Conversion

    MySQL data is converted to PostgreSQL UTF-8:

    • Source: MySQL with various character sets (latin1, utf8mb4, etc.)
    • Target: PostgreSQL UTF-8 encoding
    • Handling: Automatic conversion with error handling for invalid characters

    Text Length Considerations

    PostgreSQL handles text differently than MySQL:

    MySQL PostgreSQL Notes
    TEXT (65,535 bytes) TEXT (unlimited) No length restriction
    MEDIUMTEXT (16MB) TEXT (unlimited) No length restriction
    LONGTEXT (4GB) TEXT (unlimited) No length restriction

    JSON Data Conversion

    MySQL JSON to PostgreSQL JSONB

    JSON data is converted to JSONB for better performance:

    -- MySQL
preferences JSON

-- PostgreSQL
preferences JSONB

    Benefits of JSONB

    • Binary storage format (faster)
    • Indexing support
    • Better query performance
    • Duplicate key elimination

    Decimal and Precision Handling

    Financial Data Precision

    Financial amounts maintain precision:

    -- MySQL
amount DECIMAL(10,2)

-- PostgreSQL
amount DECIMAL(10,2)

    Currency Considerations

    For currency handling, consider using PostgreSQL's MONEY type or maintaining DECIMAL:

    -- Option 1: Maintain DECIMAL
amount DECIMAL(10,2)

-- Option 2: Use MONEY (locale-dependent)
amount MONEY

    Array and Set Conversion

    MySQL SET to PostgreSQL Array

    MySQL SET types are converted to PostgreSQL arrays:

    -- MySQL
interests SET('sports', 'music', 'travel', 'books')

-- PostgreSQL
interests TEXT[]

    Conversion Logic

    -- Split MySQL SET values into array
string_to_array(mysql_set_field, ',')

    Validation and Constraints

    Enhanced Constraints

    PostgreSQL allows more sophisticated constraints:

    -- Email validation
email VARCHAR(255) CHECK (email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$')

-- Phone number format
phone VARCHAR(20) CHECK (phone ~ '^[\d\s\-\(\)\+\.x]+$')

-- Date range validation  
birth_date DATE CHECK (birth_date BETWEEN '1900-01-01' AND CURRENT_DATE)

    NOT NULL Constraints

    Review and enhance NOT NULL constraints:

    -- Required fields
first_name VARCHAR(50) NOT NULL,
last_name VARCHAR(100) NOT NULL,
created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

    Performance Considerations

    Index Conversion

    MySQL indexes are recreated with PostgreSQL-specific optimizations:

    -- B-tree indexes (default)
CREATE INDEX idx_person_last_name ON person(last_name);

-- Partial indexes for performance
CREATE INDEX idx_active_persons ON person(last_name) WHERE is_active = true;

-- GIN indexes for arrays/JSONB
CREATE INDEX idx_person_preferences ON person USING GIN(preferences);

    Storage Optimization

    PostgreSQL storage considerations:

    • TOAST: Automatic compression for large text fields
    • Fill Factor: Adjust for high-update tables
    • Vacuum Strategy: Plan for ongoing maintenance

    Data type conversion ensures that all MySQL data is properly transformed to take advantage of PostgreSQL's advanced type system while maintaining data integrity and improving performance.
