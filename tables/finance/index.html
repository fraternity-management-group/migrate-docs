    Finance Table Migration

    Overview

    The Finance table migration transfers financial data from the MySQL database to a simplified PostgreSQL finance table. This is a direct field mapping migration with minimal transformation.

    Schema Transformation

    MySQL Source Tables

    • Finance - Financial records with gift information
    • Charge - Individual charge transactions
    • Payment - Individual payment transactions

    PostgreSQL Target Tables

    • finance - Direct mapping of finance data (one record per person)
    • transaction - Consolidated charges and payments with transaction type discrimination

    Transaction Table Integration

    Charge and Payment Consolidation

    The MySQL Charge and Payment tables are consolidated into a unified transaction table in PostgreSQL with type discrimination to distinguish between charges and payments.

    MySQL Source Schemas

    Charge Table: 

    CREATE TABLE Charge (
  id INT AUTO_INCREMENT PRIMARY KEY,
  donor_id INT NOT NULL,
  campaign_id INT NOT NULL,
  account INT NOT NULL,
  amount DOUBLE NOT NULL DEFAULT 0,
  billed TINYINT(1),
  date DATE NOT NULL,
  comments TINYTEXT,
  entered_by VARCHAR(45),
  last_changed TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

    Payment Table: 

    CREATE TABLE Payment (
  id INT AUTO_INCREMENT PRIMARY KEY,
  donor_id INT NOT NULL,
  campaign_id INT NOT NULL,
  account INT NOT NULL,
  batch INT NOT NULL DEFAULT 0,
  amount DOUBLE NOT NULL DEFAULT 0,
  post TINYINT(1),
  billed TINYINT(1),
  date DATE NOT NULL,
  entered_by VARCHAR(45),
  comments VARCHAR(255),
  thankyou TINYINT(1) NOT NULL DEFAULT 0,
  last_changed TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

    PostgreSQL Transaction Table Schema

    CREATE TABLE transaction (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  person_id UUID NOT NULL REFERENCES person(id) ON DELETE CASCADE,
  account VARCHAR(45) NOT NULL,
  type VARCHAR(20) NOT NULL CHECK (type IN ('payment', 'charge')),
  batch INTEGER,
  amount NUMERIC(18,2) NOT NULL DEFAULT 0,
  post BOOLEAN,
  billed BOOLEAN,
  date DATE NOT NULL,
  entered_by VARCHAR(45),
  comments TEXT,
  thankyou BOOLEAN,
  last_changed TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

    Transaction Migration Logic

    The migration consolidates both Charge and Payment records into the transaction table:

    // Migration handles both Charge and Payment tables
types := []struct {

    Transaction Table Benefits

    1. Unified Financial Transaction Tracking

      • Single table for all financial transactions
      • Type field distinguishes charges from payments
      • Simplified querying and reporting

    2. Enhanced Data Types

      • UUID primary keys for better distributed system support
      • NUMERIC(18,2) for precise monetary calculations
      • Native BOOLEAN types instead of TINYINT

    3. Improved Referential Integrity

      • Foreign key to person table ensures data consistency
      • Cascade deletes maintain referential integrity

    4. Flexible Transaction Management

      • Supports both charge and payment workflows
      • Batch processing support for payments
      • Thank you tracking for payments

    PostgreSQL Table Schema

    CREATE TABLE IF NOT EXISTS finance (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  person_id UUID NOT NULL REFERENCES person(id) ON DELETE CASCADE,
  gift_date DATE,
  amount NUMERIC(18,2),
  amount_owed NUMERIC(18,2),
  matching BOOLEAN,
  matching_comment VARCHAR(45),
  perpetual BOOLEAN,
  perpetual_comment VARCHAR(45),
  planned BOOLEAN,
  planned_comment VARCHAR(45),
  recognition VARCHAR(45),
  thank_you BOOLEAN,
  statements BOOLEAN,
  naming VARCHAR(45),
  last_changed TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE (person_id)
);

    donor_id person_id INT UUID Foreign key to person table
    gift_date gift_date DATE DATE Direct mapping
    amount amount DOUBLE NUMERIC(18,2) Enhanced precision
    amount_owed amount_owed DOUBLE NUMERIC(18,2) Enhanced precision
    matching matching TINYINT(1) BOOLEAN Boolean conversion
    matching_comment matching_comment VARCHAR(45) VARCHAR(45) Direct mapping
    perpetual perpetual TINYINT(1) BOOLEAN Boolean conversion
    perpetual_comment perpetual_comment VARCHAR(45) VARCHAR(45) Direct mapping
    planned planned TINYINT(1) BOOLEAN Boolean conversion
    planned_comment planned_comment VARCHAR(45) VARCHAR(45) Direct mapping
    recognition recognition VARCHAR(45) VARCHAR(45) Direct mapping
    thank_you thank_you TINYINT(1) BOOLEAN Boolean conversion
    statements statements TINYINT(1) BOOLEAN Boolean conversion
    naming naming VARCHAR(45) VARCHAR(45) Direct mapping
    last_changed last_changed TIMESTAMP TIMESTAMP Direct mapping

    Fields Not Migrated

    • campaign_id - Not included in PostgreSQL schema

    Key Constraints

    • UNIQUE constraint on person_id: Only one finance record per person
    • Foreign key to person table: Links finance data to individual records

    3. Boolean Conversion

    5. NULL values preserved

    6. Precision Enhancement

    7. DOUBLE values converted to NUMERIC(18,2) for better precision

    Key Features

    Simple Financial Tracking

    1. One Record Per Person

      • Unique constraint ensures single finance record per person
      • Consolidates financial information for each individual

    2. Basic Gift Information

      • Gift date, amount, and amount owed tracking
      • Boolean flags for matching, perpetual, and planned gifts
      • Comments for matching, perpetual, and planned giving

    3. Recognition and Communication

      • Recognition level tracking
      • Thank you and statement preferences
      • Naming rights information

    Indexes and Performance

    Primary Indexes

    -- Automatically created indexes
CREATE UNIQUE INDEX finance_pkey ON finance(id);
CREATE UNIQUE INDEX finance_person_id_key ON finance(person_id);
CREATE INDEX idx_finance_person_id ON finance(person_id);

    Common Queries

    Basic Finance Report

    SELECT
    p.first_name,
    p.last_name,
    f.amount,
    f.gift_date,
    f.recognition
FROM finance f
JOIN person p ON f.person_id = p.id
WHERE f.amount IS NOT NULL
ORDER BY f.amount DESC;

    Matching Gift Summary

    SELECT
    p.first_name,
    p.last_name,
    f.amount,
    f.matching_comment
FROM finance f
JOIN person p ON f.person_id = p.id
WHERE f.matching = true
ORDER BY f.amount DESC;

    Outstanding Amounts

    SELECT
    p.first_name,
    p.last_name,
    f.amount_owed,
    f.gift_date
FROM finance f
JOIN person p ON f.person_id = p.id
WHERE f.amount_owed > 0
ORDER BY f.amount_owed DESC;

    Transaction Analysis Queries

    Payment vs Charge Summary

    SELECT
    t.type,
    COUNT(*) as transaction_count,
    SUM(t.amount) as total_amount,
    AVG(t.amount) as average_amount
FROM transaction t
GROUP BY t.type
ORDER BY t.type;

    Person Transaction History

    SELECT
    p.first_name,
    p.last_name,
    t.type,
    t.amount,
    t.date,
    t.comments
FROM transaction t
JOIN person p ON t.person_id = p.id
WHERE p.id = $1  -- Replace with specific person UUID
ORDER BY t.date DESC;

    Outstanding Charges (Unbilled)

    SELECT
    p.first_name,
    p.last_name,
    t.amount,
    t.date,
    t.account
FROM transaction t
JOIN person p ON t.person_id = p.id
WHERE t.type = 'charge'
  AND (t.billed = false OR t.billed IS NULL)
ORDER BY t.date ASC;

    Payment Batch Summary

    SELECT
    t.batch,
    COUNT(*) as payment_count,
    SUM(t.amount) as batch_total,
    MIN(t.date) as earliest_payment,
    MAX(t.date) as latest_payment
FROM transaction t
WHERE t.type = 'payment'
  AND t.batch IS NOT NULL
GROUP BY t.batch
ORDER BY t.batch;

    Combined Finance and Transaction Report

    SELECT
    p.first_name,
    p.last_name,
    f.amount as total_gift_amount,
    f.amount_owed,
    (
        SELECT COUNT(*)
        FROM transaction t
        WHERE t.person_id = p.id AND t.type = 'charge'
    ) as total_charges,
    (
        SELECT COUNT(*)
        FROM transaction t
        WHERE t.person_id = p.id AND t.type = 'payment'
    ) as total_payments,
    (
        SELECT SUM(t.amount)
        FROM transaction t
        WHERE t.person_id = p.id AND t.type = 'payment'
    ) as total_payments_amount
FROM finance f
JOIN person p ON f.person_id = p.id
ORDER BY f.amount DESC;

    Data Quality Improvements

    Validation Rules

    1. Amount Validation

      • Amount and amount_owed use NUMERIC(18,2) for precise decimal handling
      • Currency precision limited to 2 decimal places

    2. Date Validation

      • Gift dates validated during migration
      • Last_changed timestamps preserved from source

    3. Boolean Validation

      • MySQL TINYINT(1) values properly converted to PostgreSQL BOOLEAN
      • NULL values preserved where appropriate

    Data Cleanup During Migration

    1. Person ID Validation 

      -- Verify all finance records have valid person references
SELECT COUNT(*) FROM finance f
LEFT JOIN person p ON f.person_id = p.id
WHERE p.id IS NULL;

    2. Unique Constraint Enforcement 

      -- Check for multiple finance records per person (should be 0)
SELECT person_id, COUNT(*)
FROM finance
GROUP BY person_id
HAVING COUNT(*) > 1;

    Troubleshooting

    Common Migration Issues

    1. Person Reference Errors 

      Error: foreign key constraint violation
Solution: Ensure all donor_id values from MySQL exist in person table

    2. UUID Generation Issues 

      Error: duplicate key value violates unique constraint
Solution: Check UUID generation function availability

    3. Boolean Conversion Issues 

      Error: invalid input syntax for type boolean
Solution: Verify TINYINT(1) values are 0, 1, or NULL

    4. Amount Precision Issues 

      Error: numeric field overflow
Solution: Check for amounts exceeding NUMERIC(18,2) precision

    Performance Considerations

    Optimization Strategies

    1. Finance Table Index Usage
    2. Primary key index on id for fast lookups
    3. Unique index on person_id enforces one record per person

    4. Foreign key index improves join performance with person table

    5. Transaction Table Index Recommendations 

      -- Recommended indexes for transaction table
CREATE INDEX idx_transaction_person_id ON transaction(person_id);
CREATE INDEX idx_transaction_type ON transaction(type);
CREATE INDEX idx_transaction_date ON transaction(date);
CREATE INDEX idx_transaction_account ON transaction(account);
CREATE INDEX idx_transaction_type_date ON transaction(type, date);

-- For payment batch queries
CREATE INDEX idx_transaction_batch ON transaction(batch) WHERE batch IS NOT NULL;

-- For billing status queries
CREATE INDEX idx_transaction_billed ON transaction(billed, type) WHERE billed IS NOT NULL;

    6. Query Optimization

    7. Use person_id for direct lookups rather than joins when possible
    8. Consider covering indexes for frequently accessed field combinations
    9. Monitor query performance for reporting queries involving both finance and transaction tables

    10. Use transaction type filters early in WHERE clauses

    11. Data Management

    12. Regular VACUUM and ANALYZE operations for optimal performance
    13. Consider partitioning transaction table by date for very large datasets
    14. Monitor table growth and plan for scaling
    15. Archive old transaction records based on business requirements
