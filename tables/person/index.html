    Person Table Migration

    Overview

    The person table migration is one of the most critical components of the schema transformation, as nearly every other table references person records through foreign keys.

    Schema Transformation

    MySQL Source Table: Donor

    CREATE TABLE `Donor` (
  `id` int(10) unsigned NOT NULL AUTO_INCREMENT,
  `staff` varchar(45) DEFAULT NULL,
  `full_name` varchar(255) DEFAULT NULL,
  `first_name` varchar(45) DEFAULT NULL,
  `last_name` varchar(45) NOT NULL DEFAULT '',
  `dob` date DEFAULT NULL,
  `dod` date DEFAULT NULL,
  `last_changed` timestamp NOT NULL DEFAULT current_timestamp() ON UPDATE current_timestamp(),
  `color_code` char(6) DEFAULT NULL,
  `showdir` tinyint(1) DEFAULT 1,
  `passwd` varchar(45) DEFAULT NULL,
  `maiden_name` varchar(45) DEFAULT NULL,
  `twitter` varchar(15) DEFAULT NULL,
  `linkedin` varchar(45) DEFAULT NULL,
  `facebook` varchar(45) DEFAULT NULL,
  `instagram` varchar(45) DEFAULT NULL,
  `preferred_contact_email` tinyint(1) DEFAULT 1,
  `preferred_contact_phone` tinyint(1) DEFAULT 1,
  `preferred_contact_text` tinyint(1) DEFAULT 1,
  `nickname` varchar(45) DEFAULT NULL,
  PRIMARY KEY (`id`)
);

    Note: Additional person-related fields are stored in related tables:

    • Status and person code information: General_Frat_Info table
    • Address information: Address table
    • Email addresses: Email table
    • Phone numbers: Phone table
    • Employment information: Job table
    • Family information: Family table
    • Education information: Schools table
    • Recognition information: Detail_Frat_Info table

    PostgreSQL Target Table: person

    CREATE TABLE person (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    organization_id UUID NOT NULL REFERENCES organization(id) ON DELETE CASCADE,
    staff TEXT,
    full_name TEXT,
    first_name TEXT,
    last_name TEXT NOT NULL,
    display_name TEXT NOT NULL,
    date_of_birth DATE,
    date_of_death DATE,
    color_code CHAR(6),
    show_dir BOOLEAN DEFAULT TRUE,
    maiden_name VARCHAR(45),
    nickname VARCHAR(45),
    in_chapter BOOLEAN NOT NULL DEFAULT FALSE,
    chapter TEXT,
    pin VARCHAR(45),
    initiation_date DATE,
    status person_status_enum NOT NULL DEFAULT 'Active',
    code person_code_enum NOT NULL DEFAULT 'Alumni',
    involvement_comments TEXT,
    graduation_date DATE,
    created TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    last_changed TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
    • The dob column is now date_of_birth.
    • The dod column is now date_of_death.

    Person Status

    CREATE TYPE person_status_enum AS ENUM (
    'Active',
    'In-active',
    'Lost',
    'Deceased'
);

    Maps from MySQL General_Frat_Info.mstatus:

    • 'A' 'Active'
    • 'I' 'In-active'
    • 'L' 'Lost'
    • 'D' 'Deceased'

    Person Code

    CREATE TYPE person_code_enum AS ENUM (
    'Alumni',
    'Undergrad',
    'Parent',
    'Friend',
    'Alumni/Parent'
);

    Maps from MySQL General_Frat_Info.mcode:

    • 'A' 'Alumni'
    • 'U' 'Undergrad'
    • 'P' 'Parent'
    • 'F' 'Friend'
    • 'X' 'Alumni/Parent'rving as the foundation for most other data relationships.

    Key Changes and Improvements

    1. Primary Key Modernization

    MySQL: Auto-increment integer 

    id INT AUTO_INCREMENT PRIMARY KEY

    PostgreSQL: UUID for better distributed system support 

    id UUID PRIMARY KEY DEFAULT uuid_generate_v4()

    Benefits:

    • Globally unique identifiers
    • Better for distributed systems
    • Enhanced security (non-sequential)
    • Easier data merging and synchronization

    2. Table Consolidation

    MySQL: Data spread across multiple tables

    • Basic info: Donor table
    • Status/membership: General_Frat_Info table
    • Social media: Individual fields in Donor
    • Contact preferences: Individual boolean fields in Donor

    PostgreSQL: Consolidated and normalized structure

    • Core person data: person table
    • Social media: Dedicated social_media table
    • Contact preferences: Dedicated contact_preferences table

    3. Enhanced Data Types

    Boolean Fields

    MySQL: TINYINT(1) emulation

    showdir TINYINT(1) DEFAULT 1,
in_chapter TINYINT(1) NOT NULL DEFAULT 0,
preferred_contact_email TINYINT(1) DEFAULT 1

    PostgreSQL: Native boolean type 

    show_dir BOOLEAN DEFAULT TRUE,
in_chapter BOOLEAN NOT NULL DEFAULT FALSE

    4. Organization Relationship

    PostgreSQL adds explicit organization relationship: 

    organization_id UUID NOT NULL REFERENCES organization(id) ON DELETE CASCADE

    This provides clear hierarchical structure and multi-tenancy support.

    5. Display Name Standardization

    PostgreSQL includes a computed display_name field for consistent name representation across the application.

    6. Graduation Date Addition

    PostgreSQL introduces graduation_date field: 

    graduation_date DATE

    • The person's graduation date, if available.
    • Populated from the earliest grad_date in the Schools table for each person.

    8. Data Normalization

    Social Media Extraction

    MySQL (embedded in Donor table): 

    twitter VARCHAR(15),
linkedin VARCHAR(45),
facebook VARCHAR(45),
instagram VARCHAR(45)

    PostgreSQL (normalized to separate table): 

    -- Separate social_media table
CREATE TABLE social_media (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    person_id UUID NOT NULL REFERENCES person(id) ON DELETE CASCADE,
    twitter VARCHAR(15),
    linkedin VARCHAR(45),
    facebook VARCHAR(45),
    instagram VARCHAR(45)
);

    Contact Preferences Consolidation

    MySQL (multiple boolean fields): 

    preferred_contact_email TINYINT(1) DEFAULT 1,
preferred_contact_phone TINYINT(1) DEFAULT 1,
preferred_contact_text TINYINT(1) DEFAULT 1

    PostgreSQL (structured table): 

    -- Dedicated contact_preferences table
CREATE TABLE contact_preferences (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    person_id UUID NOT NULL REFERENCES person(id) ON DELETE CASCADE,
    preferred_contact_method contact_method_enum DEFAULT 'Email',
    allows_marketing BOOLEAN NOT NULL DEFAULT TRUE,
    -- ... additional preference fields
);

    Migration Logic

    1. Data Source and Selection

    The migration extracts person data from MySQL using a complex join to ensure only valid members are migrated:

    -- Join Donor with Member and Donor_Group to get organization mapping
SELECT d.id, d.staff, d.full_name, d.first_name, d.last_name,
       d.dob, d.dod, d.last_changed, d.color_code, d.showdir,
       d.maiden_name, d.nickname, dg.id as group_id
FROM Donor d
INNER JOIN Member m ON d.id = m.donor_id
INNER JOIN Donor_Group dg ON m.group_id = dg.id
WHERE dg.main = 1

    2. UUID Generation and Mapping

    Each MySQL Donor.id is mapped to a new PostgreSQL UUID and stored for reference by other migrations:

    // Generate new UUID for each person
personID := generateUUID()
m.personMapping.Set(id, personID)  // Store mapping for other migrations

    3. Organization Relationship

    The person is linked to the correct organization using the organization mapping established during the organization migration:

    // Lookup the correct organization ID from the mapping
organizationID, exists := m.orgMapping.Get(groupID)
if !exists {
    return fmt.Errorf("failed to get organization ID for group_id %d", groupID)
}

    4. Display Name Generation

    The migration creates a computed display_name field following this priority:

    var displayName string
switch {
case fullName.Valid && fullName.String != "":
    displayName = fullName.String
case firstName.Valid && firstName.String != "" && lastName != "":
    displayName = firstName.String + " " + lastName
case lastName != "":
    displayName = lastName
case firstName.Valid && firstName.String != "":
    displayName = firstName.String
default:
    displayName = "Unknown"
}

    5. Default Values and Type Conversion

    The migration applies default values and converts MySQL types to PostgreSQL equivalents:

    // Convert TINYINT(1) to boolean
showDirBool := true
if showDir.Valid {
    showDirBool = showDir.Bool
}

// Set defaults for enum fields
status := "Active"    // Default person status
code := "Alumni"      // Default person code
inChapter := false    // Default in_chapter value

    6. Data Validation and Cleaning

    The migration includes comprehensive data validation:

    Date Validation

    // Handle nullable dates
var dobPtr *time.Time
if dob.Valid {
    dobPtr = &dob.Time
}

var dodPtr *time.Time
if dod.Valid {
    dodPtr = &dod.Time
}

    Timestamp Conversion

    // Convert MySQL TIMESTAMP from []byte to time.Time
var lastChanged time.Time
if lastChangedRaw != nil {
    t, err := time.Parse("2006-01-02 15:04:05", string(lastChangedRaw))
    if err == nil {
        lastChanged = t
    }
}

    String Field Handling

    // Handle nullable string fields
var staff, fullName, firstName, colorCode, maidenName, nickname *string
if staff.Valid { staff = &staff.String }
if fullName.Valid { fullName = &fullName.String }
// ... similar for other nullable string fields

    7. Data Normalization Process

    The migration separates embedded data into normalized tables:

    Social Media Extraction

    Social media fields from the Donor table are extracted during a separate migration step:

    • twitter social_media.twitter
    • linkedin social_media.linkedin
    • facebook social_media.facebook
    • instagram social_media.instagram

    Contact Preferences Migration

    Contact preference boolean fields are migrated to a structured contact_preferences table during a separate migration step that:

    // Determines preferred contact method based on available contact info
preferredMethod := "Email"
if !emailPersons[personID] && phonePersons[personID] {
    preferredMethod = "Phone"
}

// Inserts contact preferences with defaults
INSERT INTO contact_preferences (
    person_id, preferred_contact_method, allows_marketing,
    allows_surveys, allows_political
) VALUES (personID, preferredMethod, true, true, false)

    8. Conflict Handling and Data Integrity

    The migration uses PostgreSQL's conflict resolution to ensure data integrity:

    INSERT INTO person (...) VALUES (...)
ON CONFLICT (id) DO NOTHING

    This prevents duplicate records during migration reruns while maintaining referential integrity.

    Indexes and Performance

    Primary Indexes

    -- Primary key (automatic)
CREATE UNIQUE INDEX person_pkey ON person(id);

-- Essential lookup indexes
CREATE INDEX idx_person_last_name ON person(last_name);
CREATE INDEX idx_person_first_name ON person(first_name);
CREATE INDEX idx_person_full_name ON person(last_name, first_name);
CREATE INDEX idx_person_display_name ON person(display_name);

    Status and Code Indexes

    -- Status-based queries
CREATE INDEX idx_person_status ON person(status);
CREATE INDEX idx_person_active ON person(status) WHERE status = 'Active';

-- Person code filtering
CREATE INDEX idx_person_code ON person(code);

    Organization Relationship Index

    -- Organization membership queries
CREATE INDEX idx_person_organization ON person(organization_id);

    Composite Indexes

    -- Common query patterns
CREATE INDEX idx_person_status_code ON person(status, code);
CREATE INDEX idx_person_name_status ON person(last_name, first_name, status);
CREATE INDEX idx_person_org_status ON person(organization_id, status);

    Common Queries

    Basic Person Lookup

    -- Find person by name
SELECT id, first_name, last_name, status, code
FROM person
WHERE last_name ILIKE 'Smith%'
AND status = 'Active'
ORDER BY last_name, first_name;

    Contact Information Queries

    -- Find people with email addresses
SELECT p.first_name, p.last_name, e.email_address
FROM person p
JOIN email_address e ON p.id = e.person_id
WHERE p.status = 'Active'
AND e.is_primary = true;

-- Find people with phone numbers
SELECT p.first_name, p.last_name, ph.phone_number
FROM person p
JOIN phone_number ph ON p.id = ph.person_id
WHERE p.status = 'Active'
AND ph.is_primary = true;

    Alumni Analysis

    -- Alumni statistics by status and code
SELECT
    code,
    status,
    COUNT(*) as count
FROM person
WHERE code IN ('Alumni', 'Alumni/Parent')
GROUP BY code, status
ORDER BY code, status;

    Organization Membership

    -- People by organization
SELECT
    o.title as organization_name,
    COUNT(*) as member_count,
    COUNT(*) FILTER (WHERE p.status = 'Active') as active_members
FROM person p
JOIN organization o ON p.organization_id = o.id
GROUP BY o.id, o.title
ORDER BY member_count DESC;

    Age Demographics

    -- Age distribution for active alumni
SELECT
    CASE
        WHEN EXTRACT(YEAR FROM AGE(dob)) < 30 THEN 'Under 30'
        WHEN EXTRACT(YEAR FROM AGE(dob)) < 50 THEN '30-49'
        WHEN EXTRACT(YEAR FROM AGE(dob)) < 70 THEN '50-69'
        ELSE '70+'
    END as age_group,
    COUNT(*) as count
FROM person
WHERE status = 'Active'
AND code = 'Alumni'
AND dob IS NOT NULL
GROUP BY age_group
ORDER BY age_group;

    Data Quality Improvements

    Enhanced Validation

    The PostgreSQL schema includes enhanced validation:

    -- Name requirements
ALTER TABLE person ADD CONSTRAINT person_names_check
CHECK (
    LENGTH(TRIM(first_name)) >= 1 AND
    LENGTH(TRIM(last_name)) >= 1
);

-- Birth date validation
ALTER TABLE person ADD CONSTRAINT person_birth_date_check
CHECK (
    dob IS NULL OR
    (dob >= '1900-01-01' AND dob <= CURRENT_DATE)
);

-- Status enum validation (automatically enforced by enum type)
-- Code enum validation (automatically enforced by enum type)

    Data Standardization

    The migration includes data standardization:

    • Name Formatting: Proper handling of null/empty names with fallbacks
    • Display Name Generation: Computed field for consistent name representation
    • Boolean Type Conversion: MySQL TINYINT(1) PostgreSQL BOOLEAN
    • Date Validation: Invalid date handling with null conversion
    • Enum Type Safety: Strong typing with custom enum types

    Troubleshooting

    Common Migration Issues

    1. Missing Organization Mapping

    Issue: Person record cannot find corresponding organization Solution: Ensure organization migration completed successfully and organization mapping is available

    2. Duplicate Person Records

    Issue: Multiple donor records with same information Solution: Migration uses ON CONFLICT (id) DO NOTHING to prevent duplicates; manual cleanup may be needed

    3. Invalid Date Values

    Issue: MySQL contains invalid dates (0000-00-00) Solution: Migration converts invalid dates to NULL values

    4. Missing Display Names

    Issue: Records without proper display name generation Solution: Migration has fallback logic: full_name first+last last first "Unknown"

    5. Organization Relationship Issues

    Issue: Person records not properly linked to organizations Solution: Verify that Donor_Group.main=1 records exist and organization migration completed

    Validation Queries

    -- Check for missing required data
SELECT COUNT(*) as missing_names
FROM person
WHERE first_name IS NULL OR first_name = ''
   OR last_name IS NULL OR last_name = '';

-- Verify enum values
SELECT status, COUNT(*)
FROM person
GROUP BY status;

SELECT code, COUNT(*)
FROM person
GROUP BY code;

-- Check organization relationships
SELECT
    o.title,
    COUNT(p.id) as person_count
FROM organization o
LEFT JOIN person p ON o.id = p.organization_id
GROUP BY o.id, o.title
ORDER BY person_count DESC;

-- Verify display name consistency
SELECT COUNT(*) as missing_display_names
FROM person
WHERE display_name IS NULL OR display_name = '';
