    Prospect Table Improvements and Recommendations

    Implemented Improvements

    1. Field Name Clarity

    • loyal → is_loyal (clearer boolean naming)
    • leader → is_leader (clearer boolean naming)
    • lots_of_money → has_wealth_capacity (more professional terminology)
    • loyal_comment → loyal_assessment_comment (clearer purpose)
    • leader_comment → leader_assessment_comment (clearer purpose)
    • lots_of_money_comment → wealth_assessment_comment (clearer purpose)
    • low_priority_comment → low_priority_reason (more descriptive)
    • willingness → giving_willingness (more specific)
    • range_start/range_end → gift_range_minimum/gift_range_maximum (clearer purpose)
    • capability → giving_capability (more specific)
    • ask → suggested_ask_amount (clearer business meaning)
    • campaign_status → campaign_contact_status (more specific)
    • pkg_sent → information_package_sent_date (clearer purpose)
    • section → assigned_section (clearer purpose)
    • solicitor → assigned_solicitor (clearer purpose)
    • comments → prospect_notes (more specific)

    2. Data Integrity Constraints

    • Gift range validation (minimum ≤ maximum)
    • Positive ask amounts
    • Valid real estate counts and values
    • Valid historical gift counts

    3. Optimized Indexing Strategy

    • Wealth Analysis Index: Multi-column index for wealth scoring queries
    • High Capacity Index: Filtered index for high-capacity prospects
    • Campaign Management Index: Support for campaign status and section queries
    • Solicitor Assignment Index: Active solicitor assignments
    • Assessment Flags Index: Multi-flag assessment queries
    • Study Status Index: Campaign interaction tracking
    • Ask Amount Index: Ordered by suggested ask amounts

    4. Comprehensive Documentation

    • Table-level comments explaining purpose
    • Field-level comments explaining business meaning
    • Clear documentation of scoring systems
    • Explanation of external rating integrations

    Additional Recommendations

    1. Consider Table Normalization

    The prospect table is quite large with many disparate concerns. Consider splitting into:

    -- Core prospect information
CREATE TABLE prospect_core (
    person_id UUID PRIMARY KEY REFERENCES person(id),
    is_loyal BOOLEAN NOT NULL DEFAULT FALSE,
    is_leader BOOLEAN NOT NULL DEFAULT FALSE,
    has_wealth_capacity BOOLEAN NOT NULL DEFAULT FALSE,
    is_low_priority BOOLEAN,
    prospect_notes TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Wealth assessment data
CREATE TABLE prospect_wealth_assessment (
    person_id UUID PRIMARY KEY REFERENCES prospect_core(person_id),
    estimated_giving_capacity capacity_to_give_enum,
    wealth_rating_score wealth_score_enum,
    estimated_net_worth networth_enum,
    estimated_annual_income income_enum,
    real_estate_property_count SMALLINT DEFAULT -1,
    real_estate_total_value INTEGER DEFAULT -1,
    charitable_giving_history BOOLEAN DEFAULT FALSE,
    top_wealth_percentile_flag BOOLEAN DEFAULT FALSE,
    wealth_assessment_comment TEXT,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Campaign interaction tracking
CREATE TABLE prospect_campaign_interaction (
    person_id UUID PRIMARY KEY REFERENCES prospect_core(person_id),
    five_l_campaign_action five_l_enum,
    study_action study_action_enum,
    study_status study_status_enum,
    giving_willingness willingness_enum,
    gift_range_minimum INTEGER NOT NULL DEFAULT 0,
    gift_range_maximum INTEGER NOT NULL DEFAULT 0,
    suggested_ask_amount NUMERIC(18,2) NOT NULL DEFAULT 0,
    campaign_contact_status campaign_status_enum DEFAULT 'non_prospect',
    information_package_sent_date DATE,
    assigned_section section_enum,
    assigned_solicitor UUID REFERENCES person(id),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- External scoring systems
CREATE TABLE prospect_external_scores (
    person_id UUID PRIMARY KEY REFERENCES prospect_core(person_id),
    donor_rating_score donor_score_enum,
    donor_search_rating ds_rating_enum,
    af_algorithm_score INTEGER,
    mg_algorithm_score INTEGER,
    historical_gift_count INTEGER DEFAULT -1,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);

    Benefits: - Better separation of concerns - Improved query performance for specific data types - Easier maintenance and updates - Reduced row locking contention - Better caching strategies

    2. Add Audit Trail Support

    -- Prospect change history
CREATE TABLE prospect_audit_log (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    person_id UUID NOT NULL REFERENCES person(id),
    table_name VARCHAR(50) NOT NULL,
    field_name VARCHAR(50) NOT NULL,
    old_value TEXT,
    new_value TEXT,
    change_type VARCHAR(20) NOT NULL, -- INSERT, UPDATE, DELETE
    changed_by VARCHAR(64) NOT NULL,
    changed_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP
);

    3. Add Performance Views

    -- High-priority prospect summary
CREATE VIEW high_priority_prospects AS
SELECT
    p.person_id,
    pe.first_name,
    pe.last_name,
    p.has_wealth_capacity,
    p.giving_capability,
    p.suggested_ask_amount,
    p.campaign_contact_status,
    p.assigned_solicitor
FROM prospect p
JOIN person pe ON p.person_id = pe.id
WHERE p.has_wealth_capacity = true
   OR p.giving_capability IN ('over_100k', '50k_to_100k', '25k_to_50k')
   OR p.suggested_ask_amount >= 10000;

-- Solicitor workload summary
CREATE VIEW solicitor_workload AS
SELECT
    s.id as solicitor_id,
    s.first_name || ' ' || s.last_name as solicitor_name,
    COUNT(p.person_id) as assigned_prospect_count,
    SUM(p.suggested_ask_amount) as total_ask_amount,
    COUNT(CASE WHEN p.campaign_contact_status = 'not_contacted_yet' THEN 1 END) as pending_contacts
FROM person s
JOIN prospect p ON s.id = p.assigned_solicitor
GROUP BY s.id, s.first_name, s.last_name;

    4. Consider Materialized Views for Analytics

    -- Wealth distribution analysis (refreshed nightly)
CREATE MATERIALIZED VIEW prospect_wealth_distribution AS
SELECT
    giving_capability,
    estimated_net_worth,
    COUNT(*) as prospect_count,
    AVG(suggested_ask_amount) as avg_ask_amount,
    SUM(suggested_ask_amount) as total_potential
FROM prospect
WHERE has_wealth_capacity = true
GROUP BY giving_capability, estimated_net_worth;

-- Campaign progress summary (refreshed hourly)
CREATE MATERIALIZED VIEW campaign_progress_summary AS
SELECT
    campaign_contact_status,
    assigned_section,
    COUNT(*) as prospect_count,
    SUM(suggested_ask_amount) as total_ask_amount,
    AVG(CASE WHEN information_package_sent_date IS NOT NULL THEN 1 ELSE 0 END) as package_sent_rate
FROM prospect
GROUP BY campaign_contact_status, assigned_section;

    5. Add Better Data Validation

    -- Add enum validation functions
CREATE OR REPLACE FUNCTION validate_prospect_consistency()
RETURNS TRIGGER AS $$
BEGIN
    -- Ensure high-capacity prospects have appropriate ask amounts
    IF NEW.has_wealth_capacity = true AND NEW.suggested_ask_amount < 1000 THEN
        RAISE WARNING 'High-capacity prospect has low ask amount: %', NEW.suggested_ask_amount;
    END IF;

    -- Ensure gift ranges are reasonable
    IF NEW.gift_range_maximum > 0 AND NEW.gift_range_maximum < NEW.suggested_ask_amount THEN
        RAISE WARNING 'Ask amount exceeds gift range maximum';
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_prospect_validation
    BEFORE INSERT OR UPDATE ON prospect
    FOR EACH ROW EXECUTE FUNCTION validate_prospect_consistency();

    Migration Strategy

    If implementing these improvements:

    1. Phase 1: Implement field name changes (current)
    2. Phase 2: Add constraints and improved indexing
    3. Phase 3: Create performance views and materialized views
    4. Phase 4: Consider table normalization for new installations
    5. Phase 5: Implement audit trail and advanced validation

    Query Performance Benefits

    The improved indexing strategy supports common queries like:

    -- Find high-capacity prospects by wealth indicators
SELECT * FROM prospect
WHERE has_wealth_capacity = true
ORDER BY wealth_rating_score DESC, estimated_net_worth DESC;

-- Find prospects ready for solicitor contact
SELECT * FROM prospect
WHERE campaign_contact_status = 'not_contacted_yet'
  AND assigned_solicitor IS NOT NULL;

-- Wealth analysis for campaign planning
SELECT
    giving_capability,
    COUNT(*) as count,
    AVG(suggested_ask_amount) as avg_ask
FROM prospect
WHERE has_wealth_capacity = true
GROUP BY giving_capability;

    These queries now benefit from targeted indexes and clearer field semantics.
