    Troubleshooting

    This guide helps you diagnose and resolve common issues when using the GMS Database Migration Tool.

    Quick Diagnosis

    Check Migration Status

    # Check if migration tool is working
./gms-migrate --version

# Test database migration
./gms-migrate validate

    Common Error Patterns

    Error Pattern Likely Cause Quick Fix
    connection refused Database not running or network issue Check database status and connectivity
    authentication failed Wrong credentials Verify username/password
    database does not exist Target database not created Create database and apply schema
    foreign key constraint Missing referenced records Check migration order and data integrity
    out of memory Insufficient system memory Reduce batch size
    timeout Network latency or database load Increase timeout values

    Database Connection Issues

    MySQL Connection Problems

    Problem: Connection Refused

    Error: dial tcp 127.0.0.1:3306: connect: connection refused

    Diagnosis Steps: 

    # Check if MySQL is running
sudo systemctl status mysql
# or on macOS
brew services list | grep mysql

# Test direct connection
mysql -h localhost -u username -p database_name

# Check MySQL port
netstat -tuln | grep 3306

    Solutions: 1. Start MySQL Service 

    # Linux
sudo systemctl start mysql

# macOS
brew services start mysql

# Windows
net start mysql

    1. Check MySQL Configuration 

      # Check MySQL config
mysql --help | grep -A 1 "Default options"

# Verify bind-address in my.cnf
grep bind-address /etc/mysql/mysql.conf.d/mysqld.cnf

    2. Firewall Issues 

      # Check firewall (Linux)
sudo ufw status
sudo ufw allow 3306

# Check firewall (macOS)
sudo pfctl -sr | grep 3306

    Problem: Authentication Failed

    Error: Access denied for user 'username'@'host' (using password: YES)

    Solutions: 1. Verify Credentials 

    # Test credentials directly
mysql -h hostname -u username -p database_name

    1. Check User Permissions 

      -- In MySQL as root
SELECT user, host FROM mysql.user WHERE user = 'your_username';
SHOW GRANTS FOR 'your_username'@'your_host';

-- Grant necessary permissions
GRANT SELECT ON database_name.* TO 'username'@'host';
FLUSH PRIVILEGES;

    2. Password Issues 

      -- Reset password
ALTER USER 'username'@'host' IDENTIFIED BY 'new_password';
FLUSH PRIVILEGES;

    PostgreSQL Connection Problems

    Problem: Connection Refused

    Error: dial tcp 127.0.0.1:5432: connect: connection refused

    Diagnosis Steps: 

    # Check if PostgreSQL is running
sudo systemctl status postgresql
# or on macOS
brew services list | grep postgresql

# Test direct connection
psql -h localhost -U username -d database_name

# Check PostgreSQL port
netstat -tuln | grep 5432

    Solutions: 1. Start PostgreSQL Service 

    # Linux
sudo systemctl start postgresql

# macOS
brew services start postgresql

# Windows
net start postgresql

    1. Check PostgreSQL Configuration 
      # Find config file
sudo -u postgres psql -c "SHOW config_file;"

# Check listen_addresses
grep listen_addresses /etc/postgresql/*/main/postgresql.conf

# Check pg_hba.conf for authentication
sudo -u postgres cat /etc/postgresql/*/main/pg_hba.conf

    Problem: Database Does Not Exist

    Error: database "gms" does not exist

    Solutions: 1. Create Database 

    # Create database
createdb -U postgres gms

# Or via psql
psql -U postgres -c "CREATE DATABASE gms;"

    1. Apply Schema 
      # Apply migration files
psql -U postgres -d gms -f db/migrations/000001_initial.up.sql
psql -U postgres -d gms -f db/migrations/000002_contact_log.up.sql

    Problem: SSL/TLS Issues

    Error: SSL is not enabled on the server

    Solutions: 1. Disable SSL for Development 

    # Update connection string
export POSTGRES_DSN="postgres://user:pass@host:5432/db?sslmode=disable"

    1. Configure SSL (Production) 
      # In postgresql.conf
ssl = on
ssl_cert_file = 'path/to/cert.pem'
ssl_key_file = 'path/to/key.pem'

    Migration Runtime Issues

    Memory Issues

    Problem: Out of Memory

    Error: runtime: out of memory: cannot allocate

    Solutions: 1. Close Other Applications 

    # Check memory usage
free -h
top -o %MEM

# Monitor migration memory usage
top -p $(pgrep gms-migrate)

    1. Increase System Memory
    2. Add more RAM to the system
    3. Use a machine with more memory
    4. Consider running migration on a dedicated server

    Problem: Database Memory Issues

    Error: server closed the connection unexpectedly

    Solutions: 1. PostgreSQL Memory Configuration 

    # In postgresql.conf
shared_buffers = 256MB
work_mem = 8MB
maintenance_work_mem = 64MB
effective_cache_size = 1GB

    1. MySQL Memory Configuration 
      # In my.cnf
innodb_buffer_pool_size = 512M
sort_buffer_size = 2M
read_buffer_size = 2M

    Performance Issues

    Problem: Migration Running Too Slowly

    Migration progress: 2% after 30 minutes

    Diagnosis: 

    # Check system resources
htop
iostat -x 1
iotop

# Check database performance
# PostgreSQL
SELECT * FROM pg_stat_activity;

# MySQL
SHOW PROCESSLIST;

    Solutions: 1. Database Optimization 

    -- PostgreSQL: Temporarily disable autovacuum
ALTER SYSTEM SET autovacuum = off;
SELECT pg_reload_conf();

-- Re-enable after migration
ALTER SYSTEM SET autovacuum = on;
SELECT pg_reload_conf();

    Problem: Network Timeouts

    Error: context deadline exceeded

    Solutions: 1. Network Optimization 

    # Check network latency
ping database_host

# Use local network when possible
# Avoid running over WAN connections

    Data Integrity Issues

    Problem: Foreign Key Violations

    Error: insert or update on table "address" violates foreign key constraint

    Diagnosis: 

    -- Check for orphaned records
SELECT COUNT(*) FROM address a
LEFT JOIN person p ON a.person_id = p.id
WHERE p.id IS NULL;

    Solutions: 1. Manual Data Cleanup 

    -- Remove orphaned records from source
DELETE FROM mysql.Address
WHERE donor_id NOT IN (SELECT donor_id FROM mysql.Donor);

    Problem: Data Type Conversion Errors

    Error: invalid input value for enum person_status_enum: "Unknown"

    Solutions: 1. Check Source Data 

    -- Find invalid enum values
SELECT DISTINCT status FROM mysql.Donor
WHERE status NOT IN ('Active', 'In-active', 'Lost', 'Deceased');

    1. Add Custom Mapping 
      # The migration tool should handle this automatically
# Check logs for mapping details
./gms-migrate migrate --verbose --log-level debug

    Configuration Issues

    Problem: Configuration File Not Found

    Error: config file not found: .gms-migrate.toml

    Solutions: 1. Create Configuration File 

    # Create basic config
cat > gms-migrate.toml << EOF
[mysql]
dsn = "user:pass@tcp(localhost:3306)/fmg"

[postgres]
dsn = "postgres://user:pass@localhost:5432/gms"

EOF

    1. Use Environment Variables 

      # Set required environment variables
export MYSQL_DSN="user:pass@tcp(localhost:3306)/fmg"
export POSTGRES_DSN="postgres://user:pass@localhost:5432/gms"

# Run without config file
./gms-migrate migrate

    2. Specify Custom Config Path 

      # Use custom config location
./gms-migrate migrate --config /path/to/config.toml

    Problem: Environment Variables Not Working

    Error: required configuration missing

    Solutions: 1. Verify Environment Variables 

    # Check if variables are set
env | grep -E "(MYSQL|POSTGRES)"

# Set missing variables
export MYSQL_DSN="your_mysql_connection"
export POSTGRES_DSN="your_postgres_connection"

    1. Use .env File 
      # Create .env file
cat > .env << EOF
MYSQL_DSN=user:pass@tcp(localhost:3306)/fmg
POSTGRES_DSN=postgres://user:pass@localhost:5432/gms
EOF

# Source the file
source .env
./gms-migrate migrate

    Schema Issues

    Problem: Table Does Not Exist

    Error: relation "person" does not exist

    Solutions: 1. Apply Schema Migrations 

    # Apply PostgreSQL schema
psql -d gms -f db/migrations/000001_initial.up.sql
psql -d gms -f db/migrations/000002_contact_log.up.sql

    1. Verify Schema Application 
      -- Check if tables exist
\dt

-- Check specific table
\d person

    Problem: Column Does Not Exist

    Error: column "contact_preferences" of relation "person" does not exist

    Solutions: 1. Check Schema Version 

    -- Check applied migrations
SELECT * FROM schema_migrations;

    1. Manually Add Missing Columns 
      -- Add missing column
ALTER TABLE person ADD COLUMN contact_preferences JSONB;

    Recovery and Rollback

    Problem: Migration Partially Completed

    Migration stopped at step 15/23

    Solutions: 1. Resume from Checkpoint 

    # Resume migration
./gms-migrate migrate --resume

# Check checkpoint status
cat migration_checkpoint.json

    1. Start Specific Step 
      # Start from specific step
./gms-migrate migrate --start-from-step 16

    Problem: Need to Rollback Migration

    Migration completed but data is incorrect

    Solutions: 1. Restore from Backup 

    # Restore PostgreSQL from backup
pg_restore -d gms backup_before_migration.sql

# Or from SQL dump
psql gms < backup_before_migration.sql

    1. Partial Rollback 
      # Clear specific tables and re-migrate
psql gms -c "TRUNCATE TABLE contact_log CASCADE;"
./gms-migrate migrate --start-from-step 23

    Debugging Tools

    Database Query Logging

    PostgreSQL

    -- Enable query logging
ALTER SYSTEM SET log_statement = 'all';
SELECT pg_reload_conf();

-- Check logs
tail -f /var/log/postgresql/postgresql-*-main.log

    MySQL

    -- Enable query logging
SET GLOBAL general_log = 'ON';
SET GLOBAL log_output = 'table';

-- View query log
SELECT * FROM mysql.general_log
WHERE command_type = 'Query'
ORDER BY event_time DESC
LIMIT 20;

    System Monitoring

    # Monitor system resources
htop

# Monitor disk I/O
iotop -o

# Monitor network
iftop

# Database-specific monitoring
pg_top              # PostgreSQL
mytop              # MySQL

    Getting Help

    Information to Collect

    When seeking help, provide:

    1. Error Message: Complete error output
    2. Configuration: Sanitized config file or environment variables
    3. Migration Step: Which step failed
    4. System Info: OS, memory, disk space
    5. Database Versions: MySQL and PostgreSQL versions
    6. Migration Logs: Relevant log entries

    Most migration issues can be resolved by checking database connectivity, verifying configuration, and ensuring proper schema setup. When in doubt, start with the validation command and work through issues systematically.
